## 1. システム概要

### 1.1 目的
AI仙人のメソドロジーに基づく、AI搭載FX分析・ペーパートレードシステム。
マルチAIコンセンサス（Gemini + Groq + アルゴリズム）による3層投票で、
高精度なエントリー・エグジット判断を自動化する。

### 1.2 システム名称
**AI FX Trading Tool** - AI仙人メソドロジー自動売買システム

### 1.3 動作モード（RunMode）

| モード | 説明 | エントリー |
|--------|------|-----------|
| **PAPER** | OANDA Practice口座でシミュレーション取引（デフォルト） | 可 |
| **BACKTEST** | 過去データに基づくストラテジー検証 | N/A |
| **ANALYSIS** | 特定ペアのシグナル分析のみ | 不可 |
| **ANALYSIS_ONLY** | OANDA主系価格不達時の自動遷移 | 不可 |
| **SAFE_STOP** | DD超過/DB障害/致命的不整合での緊急停止 | 不可 |

#### モード遷移

```text
PAPER
  ├→ ANALYSIS_ONLY   : OANDA主系価格不達
  ├→ SAFE_STOP       : DD超過 / DB障害 / 致命的不整合
ANALYSIS_ONLY
  ├→ PAPER           : OANDA復旧
  └→ SAFE_STOP       : 致命的障害
```

### 1.4 フェーズ制御（Phase）

リスク管理の段階的有効化を制御する。

| フェーズ | リスク制御 | 用途 |
|---------|-----------|------|
| **Phase 1** | 固定0.5%、適応的リスクはshadow計算のみ | フォワードテスト初期 |
| **Phase 2** | 連敗係数 + DD係数を適用 | 安定運用確認後 |
| **Phase 3** | 全適応要因（レジーム + 勝率）を適用 | 本番運用 |

### 2.3 シグナル生成パイプライン

```
1. パターン検出（アルゴリズム）
   ↓
2. テクニカル指標合流確認
   ↓
3. AIビジュアル分析（Gemini） ← 4H足確定時のみ、予算管理下
   ↓
4. 方向性バイアス確認（Groq） ← 4H足確定時のみリフレッシュ
   ↓
5. マルチAIコンセンサス投票（Algorithm主導 + 2/3多数決）
   ↓
6. リスク検証（Phase別適応制御） → エントリー実行
```

### 2.6 マルチAIコンセンサスエンジン

#### 2.6.1 投票メンバー
| # | ソース | 役割 | 出力 |
|---|--------|------|------|
| 1 | Algorithm | パターン検出（**ゲートキーパー**） | BUY / SELL / NONE |
| 2 | Gemini | チャート画像 + マクロ分析 | BUY / SELL / NONE |
| 3 | Groq | レジーム + センチメント | BUY / SELL / NEUTRAL |

#### 2.6.2 エントリー判定ルール（標準3者モード）
- **Algorithm がゲートキーパー**: Algorithm が BUY または SELL を出さない限り判定しない
- **2/3多数決**: 3投票者中2名以上が同方向
- **Algorithm が多数派に含まれる**ことが必須
- コンセンサス成立時、合意者の信頼度を平均化（閾値: **55%以上**）

#### 2.6.3 2者フォールバックモード
Gemini が利用不可（予算枯渇/API障害/非4H確定時）の場合に発動:
- **Algorithm と Groq が同方向**に投票
- 平均信頼度 **≥ 62%**（標準より厳格）
- `mode: FALLBACK_TWO_VOTE` として記録
- フォールバック発動理由を必ず保存

#### 2.6.4 クローズ判定ルール
- **Phase 1**: ログ記録のみ（実際のクローズは行わない）
- **Phase 2以降**:
  - Gemini と Groq の**両方が保有方向と反対**に投票
  - 両者の信頼度 **≥ 80%**
  - Geminiの分析鮮度が **20分以内**
  - 条件未達 → ポジション保持

### 2.7 3層分析アーキテクチャ

| レイヤー | タイミング | 分析内容 | API使用 | エントリー |
|---------|-----------|---------|---------|-----------|
| Layer 0 | 毎5分+10秒 | アルゴリズム + Groq（キャッシュ） | Groq: キャッシュのみ | 不可 |
| Layer 1 | 4H足確定+15秒 | 全分析（Gemini + Groq + アルゴ） | Gemini: 予算管理下, Groq: リフレッシュ | **可** |
| Layer 2 | 毎5分+40秒 | ポジション再評価（Groq + アルゴ） | Groq: キャッシュ + Geminiキャッシュ | 不可 |

#### 4H足確定時刻（UTC）
`00:00`, `04:00`, `08:00`, `12:00`, `16:00`, `20:00`

#### エントリーゲート
- エントリーは **Layer 1（4H足確定時）のみ** 許可
- Layer 0サイクルでは分析のみ実行（ダッシュボード表示用）
- Layer 2はポジション管理のみ（トレーリングストップ、AIクローズ判定）

### 2.8 Gemini API予算管理

| 項目 | 仕様 |
|------|------|
| 日次ソフト上限 | 16回 |
| 日次ハード上限 | 18回（予備2回） |
| 4Hサイクル上限 | 3回/サイクル |
| 呼び出し条件 | 4H足確定 AND アルゴがパターン検出済み |
| 候補選抜 | `pattern_score` 降順で優先度付け |
| キャッシュ | ペアごとの最終分析結果を保持 |
| 日次リセット | 日付変更時にカウンターをゼロリセット |
| モデル | gemini-2.5-flash |

#### スキップ理由（GeminiSkipReason）
| 理由 | 説明 |
|------|------|
| `NOT_H4_CLOSE` | 4H足確定時ではない |
| `NO_PATTERN` | アルゴリズムがパターン未検出 |
| `SCORE_TOO_LOW` | PatternScore が閾値未満 |
| `BUDGET_EXHAUSTED` | 日次上限到達 |
| `NOT_SELECTED_BY_PRIORITY` | 優先度選抜から外れた |
| `API_ERROR` | API呼び出しエラー |

### 2.9 Groq API最適化

| 項目 | 仕様 |
|------|------|
| 方向性投票 | 4H足確定時のみリフレッシュ、それ以外はキャッシュ |

## 3. リスク管理仕様

### 3.1 基本パラメータ

| パラメータ | 値 | 説明 |
|-----------|-----|------|
| 1トレードリスク | 口座の1%（Phase 1は固定0.5%） | `risk_per_trade_pct` |
| ストップロス | ATR × 2.0 | `stop_loss_atr_multiple` |
| テイクプロフィット | R:R 2:1 | `take_profit_rr_ratio` |
| 最小R:R | 0.5 | `min_risk_reward_ratio` |
| 最大ドローダウン | 10% → 全停止 | `max_drawdown_pct` |
| デイリー損失上限 | 2% | `max_daily_loss_pct` |
| ウィークリー損失上限 | 5% | `max_weekly_loss_pct` |
| 最大同時ポジション | 6（動的: 3-6） | `max_open_positions` |
| ペア別最大ポジション | 2 | `max_positions_per_pair` |
| トレーリングストップ | ATR × 1.5 | `trailing_stop_atr_multiple` |

### 3.2 適応的リスク制御（AI駆動）

```
最終リスク = 基本リスク(1%) × 連敗係数 × DD係数 × レジーム係数 × 勝率係数
```

| 要因 | 動作 | 設定 | Phase |
|------|------|------|-------|
| 連敗スケーリング | 連敗ごとにリスク×0.75（最低25%） | `loss_scale_factor: 0.75` | Phase 2+ |
| ドローダウン防御 | DD 3%→×0.75, 5%→×0.50, 7%→×0.25 | `drawdown_thresholds` | Phase 2+ |
| AI市場レジーム | Geminiがトレンド/レンジ/高ボラ判定 | AI自動判定 | Phase 3 |
| 勝率モメンタム | 直近20トレード基準 | `performance_window: 20` | Phase 3 |
| パターン無効化 | 勝率25%未満のパターンをスキップ | `pattern_disable_win_rate: 0.25` | Phase 2+ |
| 強制クールダウン | 5連敗で一時停止 | `max_consecutive_losses: 5` | Phase 2+ |

### 3.3 動的ポジション上限

| リスク倍率 | 最大ポジション数 | 状態 |
|-----------|----------------|------|
| ≥ 0.8 | 6 | 好調時 |
| ≥ 0.5 | 5 | 通常時 |
| ≥ 0.3 | 4 | やや不調 |
| < 0.3 | 3 | 不調時 |

### 3.4 通貨エクスポージャー制御

| 項目 | 値 |
|------|-----|
| 同一通貨最大エクスポージャー | 3ポジション |
| 判定方法 | 通貨ペアの構成通貨ごとにカウント |

### 3.5 信頼度閾値の自動調整

| 項目 | 値 |
|------|-----|
| 初期値 | 55% |
| 下限 | 50% |
| 上限 | 65% |
| 調整幅 | ±5%/週 |
| 判定基準 | 直近20トレードのPFに基づきルールベース判定 → Groq AIが妥当性確認 |

## 8. 永続化仕様

### 8.1 ポジションストア（SQLite）
- ファイル: `data/positions.db`
- オープンポジションの永続化（再起動復元対応）
